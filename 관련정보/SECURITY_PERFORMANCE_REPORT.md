## 1. 작업 배경

MyBob 앱은 기능 개발 위주로 빠르게 만들어지면서 보안과 성능에 신경 쓸 여유가 없었습니다.  
이번 작업은 실제 코드를 전수 분석한 후, 문제가 확인된 항목만 골라서 수정했습니다.

> **원칙**: 추측으로 수정하지 않음. 코드에서 직접 확인된 것만 고침.

## 2. A. 보안 개선

### A-1. 에러 메시지 정보 노출 차단 (전체 API)

#### 문제가 뭐였나?
앱에서 서버로 요청을 보낼 때, 서버 내부에서 오류가 나면 그 오류 내용을 그대로 사용자에게 돌려보내고 있었습니다.

**수정 전 코드 예시:**
```
// 오류 발생 시
return { error: error.message }  // "relation 'meals' does not exist" 같은 내부 오류가 그대로 전달
```

**수정 후:**
```
// 오류 발생 시
console.error('[meals GET]', error.message)  // 서버 로그에만 기록
return { error: '서버 오류가 발생했습니다.' }  // 사용자에게는 이것만 전달
```

#### 왜 위험했나?
해커는 에러 메시지를 통해 앱 내부 구조를 파악합니다.  
예를 들어 "relation 'profiles' does not exist" 같은 메시지가 노출되면,  
- 어떤 테이블이 있는지
- 어떤 컬럼이 있는지
- 어떤 DB를 쓰는지

이런 정보를 알게 되고, 이것을 바탕으로 더 정교한 공격을 설계할 수 있습니다.

#### 수정된 파일
`api/meals`, `api/friends`, `api/profile`, `api/recommendation`, `api/diagnosis`, `api/admin/meals`, `api/admin/users`, `api/admin/stats`, `api/admin/reports` — 9개 파일 전체

---

### A-2. 로그인 없이 AI 무료 사용 가능 구멍 차단

#### 문제가 뭐였나?
`/api/recommendation` (홈 코치 코멘트)과 `/api/diagnosis` (AI 진단) 두 API가  
**로그인 여부를 확인하지 않고** Gemini AI를 호출하고 있었습니다.

**수정 전:**
```
// /api/recommendation
export async function POST(request) {
  const { today, weekly, goal } = await request.json();
  // ← 여기서 바로 Gemini 호출. 누구나 가능.
  const response = await fetch(geminiUrl, ...)
}
```

**수정 후:**
```
export async function POST(request) {
  const user = await getUser(request);  // 로그인 확인 먼저
  if (!user) return { error: 'Unauthorized', status: 401 };
  
  const { today, weekly, goal } = await request.json();
  // 이제 로그인한 사용자만 Gemini 호출 가능
}
```

#### 왜 위험했나?
Gemini API는 호출 1건마다 비용이 발생합니다.  
악의적인 사람이 이 API 주소를 알아내면, 스크립트로 수천 번 자동 호출해서  
MyBob의 Gemini API 비용을 폭증시킬 수 있었습니다.  
또한 일일 사용 한도(무료 10회/PRO 25회)도 로그인 없이는 체크 자체가 안 됐습니다.

### A-3. 이미지 크기 제한 추가

#### 문제가 뭐였나?
식단 사진을 업로드할 때, **파일 크기 제한이 없었습니다.**  
누군가 100MB짜리 파일을 올리면 서버가 그대로 처리하려 했습니다.

**수정 후:**
```
// base64 데이터 길이로 실제 파일 크기를 계산
const estimatedBytes = Math.ceil(base64Data.length * 0.75);
if (estimatedBytes > 5 * 1024 * 1024) {  // 5MB 초과 시
  return { error: '이미지 크기는 5MB를 초과할 수 없습니다.', status: 400 };
}
```

#### 왜 5MB인가?
모바일에서 찍은 일반 음식 사진은 1~4MB 수준입니다.  
5MB면 충분히 수용하면서도 비정상적으로 큰 파일은 차단할 수 있습니다.

#### base64란?
이미지를 텍스트 형태로 변환한 것입니다. 실제 파일보다 약 33% 크기가 커집니다.  
그래서 "base64 길이 × 0.75 = 실제 파일 크기"로 계산합니다.

### A-4. 관리자 검색 특수문자 처리

#### 문제가 뭐였나?
관리자 페이지에서 식단을 검색할 때, 검색어를 DB에 그대로 넣고 있었습니다.

**수정 전:**
```
query.ilike('food_name', `%${search}%`)
// search에 '%' 문자가 들어오면 DB 전체 검색 패턴이 바뀜
```

**수정 후:**
```
const escaped = search.replace(/[\\%_]/g, '\\$&');  // 특수문자를 무력화
query.ilike('food_name', `%${escaped}%`)
```

#### `%`, `_`가 왜 위험한가?
SQL에서 `%`는 "뭐든 OK"를 의미하는 와일드카드입니다.  
`_`는 "딱 1글자 뭐든 OK"입니다.  
검색어에 이것을 넣으면 의도치 않은 범위의 데이터가 조회될 수 있습니다.  
이스케이프 처리로 이 문자들을 일반 글자로 취급하도록 했습니다.

## 3. B. 성능 개선 (빠른 효과)

### B-1. 홈 화면 AI 코치 중복 호출 제거

#### 문제가 뭐였나?
홈 화면에서 AI 코치 코멘트를 가져오는 로직이 **최대 2번 실행될 수 있었습니다.**

**수정 전 흐름:**
```
1. 앱 실행
2. localStorage(폰 내부 저장소)에서 오늘 식단 읽기
3. 오늘 기록이 5개 이상이면 → AI 코치 API 호출 (1번째)
4. 서버에서 최신 식단 받아오기 (동기화)
5. 동기화 후 기록이 5개 이상이면 → AI 코치 API 또 호출 (2번째!) ← 문제
```

이미 3번에서 AI를 불렀는데, 5번에서 `aiFetchedRef.current`가 `true`가 아닌 경우  
(타이밍 문제로 인해) 또 부를 수 있었습니다.

**수정 후 흐름:**
```
1. 앱 실행
2. localStorage에서 식단 읽어서 화면 즉시 표시
3. 서버 동기화 (백그라운드)
4. 동기화 완료 후, 최종 데이터로 AI 코치 1번만 호출
```

AI를 서버 동기화 완료 후에만 호출하도록 순서를 바꿔서 중복을 완전히 차단했습니다.

#### 효과
- Gemini API 호출 횟수 최대 50% 감소 → 비용 절감
- 홈 화면 응답 속도 개선

### B-2. 음식 분석 병렬 처리

#### 문제가 뭐였나?
음식 사진을 분석할 때 내부적으로 3단계를 거칩니다:
1. 음식 이름 인식 (Gemini 빠른 텍스트 호출)
2. 영양 DB 조회 (한식 DB 또는 OpenFoodFacts)
3. Gemini 비전 분석 (실제 영양소 계산)

**수정 전:** 2번과 3번을 순서대로 실행
```
한식DB 조회 → 없으면 OpenFoodFacts 조회 → 완료 후 Gemini 분석 시작
[시간: 한식DB(0.01초) + OpenFoodFacts(1~2초) + Gemini(2~3초) = 최대 5초]
```

**수정 후:** 2번과 3번을 동시에 실행
```
OpenFoodFacts 조회  ─┐
                      ├→ 둘 다 완료되면 결과 합치기
Gemini 분석        ─┘
[시간: max(OpenFoodFacts 1~2초, Gemini 2~3초) = 최대 3초]
```

#### Promise.all이란?
여러 작업을 "동시에" 시작하고, 전부 끝날 때까지 기다리는 방식입니다.  
마치 짜장면과 탕수육을 동시에 주문하는 것과 같습니다.  
순서대로 주문하면 짜장면 나오고 + 탕수육 나오는 시간이지만,  
동시에 주문하면 둘 중 더 오래 걸리는 것의 시간만 기다리면 됩니다.

#### 효과
- 한식 DB에 없는 음식(OpenFoodFacts 조회 발생 시) 분석 시간 1~2초 단축

### B-3. 사용하지 않는 바코드 라이브러리 제거

#### 문제가 뭐였나?
초기에 바코드 스캔 기능을 개발하다 중단했는데,  
사용하지 않는 라이브러리가 `package.json`에 그대로 남아 있었습니다.

```json
"@zxing/browser": "^0.2.0",   // 바코드 스캔 라이브러리
"@zxing/library": "^0.22.0",  // 바코드 스캔 라이브러리
```

코드에서도 주석 처리만 되어 있었습니다:
```
// import { BrowserMultiFormatReader } from '@zxing/browser'; // TODO: 바코드 스캔 재작업 시 복구
// import { NotFoundException } from '@zxing/library';
```

#### 왜 문제인가?
패키지가 설치되어 있으면 Next.js가 빌드할 때 번들(앱 파일 묶음)에 포함할지 검토합니다.  
실제로 포함되지 않더라도 빌드 시간이 늘어나고, 의존성 보안 취약점 검사 대상에 포함됩니다.  
사용하지 않는 라이브러리는 깔끔하게 제거하는 것이 원칙입니다.

#### 제거된 패키지
- `@zxing/browser` (~100KB)
- `@zxing/library` (~200KB)
- `@zxing/text-encoding` (의존성)

> 바코드 스캔 기능은 나중에 재개발할 때 다시 설치하면 됩니다.

## 4. C. 성능 개선 (구조 개선)

### C-1. 기록 페이지 무한스크롤

#### 문제가 뭐였나?
기록(히스토리) 페이지를 열면 **모든 식단 기록을 한 번에 화면에 그렸습니다.**  
500개 기록이 있다면 500개의 카드를 전부 만들어서 메모리에 올렸습니다.

이것은 두 가지 문제를 일으킵니다:
1. **초기 로딩이 느림** — 500개를 전부 준비해야 화면이 나타남
2. **스크롤이 버벅임** — 브라우저가 보이지도 않는 500개를 계속 관리해야 함

#### 수정 후: 무한스크롤
```
처음에는 20개만 표시
→ 스크롤을 내려서 화면 하단 근처에 오면
→ 자동으로 20개 더 추가
→ 끝에 도달할 때까지 반복
```

#### IntersectionObserver란?
브라우저에 내장된 기능으로, "이 요소가 화면에 들어왔나?"를 감지합니다.  
목록 맨 아래에 보이지 않는 작은 `div`(sentinel)를 두고,  
이 div가 화면에 보이기 시작하면 "다음 20개를 더 불러와"라고 신호를 보내는 방식입니다.

```
[식단 카드 1~20]
[식단 카드 21~40] ← 스크롤하면서 여기까지 오면
[sentinel div]   ← 이게 화면에 들어오는 순간, 다음 20개 로드
```

#### 추가 개선: 필터 변경 시 리셋
검색어나 카테고리를 바꾸면 자동으로 처음 20개부터 다시 보여줍니다.  
"짜장면"으로 검색하면 맨 위부터 보여주는 것이 당연하니까요.

### C-2. 이미지 최적화 (Next.js Image)

#### 문제가 뭐였나?
기존에는 이미지를 보여줄 때 일반 HTML `<img>` 태그를 사용했습니다.

```html
<img src="https://xxx.supabase.co/storage/.../photo.jpg" />
```

이 방식은 **원본 사진을 그대로 다운로드**합니다.  
모바일에서 찍은 사진은 보통 3~5MB인데, 기록 목록에서 보여주는 작은 썸네일에도  
3~5MB 원본을 전부 다운로드하고 있었습니다.

#### 수정 후: Next.js Image 컴포넌트

Next.js에서 제공하는 `Image` 컴포넌트로 교체했습니다.  
이 컴포넌트는 다음을 자동으로 처리합니다:

| 기능 | 설명 |
|------|------|
| **WebP 변환** | JPG/PNG를 WebP 포맷으로 자동 변환 (같은 품질에서 30~50% 용량 감소) |
| **사이즈 최적화** | 화면 크기에 맞는 작은 버전으로 자동 리사이징 |
| **Lazy Loading** | 화면에 보이는 이미지만 다운로드 |
| **캐싱** | 한 번 받은 이미지는 브라우저가 기억 |

#### 로컬 저장 이미지는?
`local:1234567890` 형태로 저장된 이미지(폰 내부 저장)는  
IndexedDB에서 base64 데이터로 꺼내오는데,  
Next.js Image는 이런 데이터 형식을 지원하지 않습니다.  
그래서 **Supabase URL이면 Next.js Image, 로컬 저장이면 기존 `<img>`**로 자동 분기하도록 했습니다.

```
photoUrl이 "https://"로 시작하면 → Next.js Image (최적화 적용)
photoUrl이 "local:..."이면       → 일반 <img> (기존 방식 유지)
```

#### `MealPhoto` 컴포넌트란?
앱 전체에서 식단 사진을 표시하는 공통 컴포넌트입니다.  
히스토리 목록, 상세 페이지 등 사진이 나오는 모든 곳에서 이 컴포넌트를 사용합니다.  
여기 한 곳만 수정하면 앱 전체에 적용됩니다.

### C-3. 레이아웃 로그인 체크 최적화

#### 문제가 뭐였나?
앱의 전체 레이아웃(`layout.tsx`)에서 로그인 여부를 확인하는 코드가 있습니다.  
이 코드가 **페이지를 이동할 때마다 다시 실행되고 있었습니다.**

**수정 전 동작:**
```
홈 → 기록 페이지로 이동
  → layout.tsx가 "pathname이 바뀌었다"고 감지
  → 로그인 상태 다시 확인 (서버 통신)
  → 기록 → 리포트로 이동
  → layout.tsx가 또 "pathname이 바뀌었다"고 감지
  → 로그인 상태 또 확인
  → ... (이동할 때마다 반복)
```

#### 왜 이렇게 됐나?
React의 `useEffect`는 의존성 배열(`[]` 안에 적는 값)이 바뀔 때마다 다시 실행됩니다.  
기존 코드에서 의존성 배열에 `pathname`(현재 페이지 주소)이 포함되어 있어서,  
페이지가 바뀔 때마다 전체 로직이 다시 실행됐습니다.

**수정 후 동작:**
```
앱 첫 실행 시 1번만 로그인 상태 확인
→ 이후 로그인/로그아웃 이벤트가 발생할 때만 재확인
→ 단순 페이지 이동 시에는 체크 없음
```

#### `onAuthStateChange`란?
Supabase에서 제공하는 이벤트 리스너입니다.  
실제로 로그인/로그아웃이 일어날 때만 신호를 보내줍니다.  
페이지만 이동하는 건 로그인 상태와 무관하므로 아무 신호도 오지 않습니다.  
이걸 활용해서 "불필요한 체크는 안 하고, 정말 필요할 때만 체크"하도록 바꿨습니다.

#### `useRef`로 pathname 추적
다만 로그인/로그아웃 시 "지금 어느 페이지에 있냐"는 알아야 합니다.  
(보호된 페이지면 로그인 페이지로 보내야 하니까)  
`useRef`로 현재 pathname을 항상 최신값으로 유지하면서,  
auth 체크 로직은 재실행하지 않도록 분리했습니다.
